# 05 — MoonBit Architecture

## Proposed packages

Initial project generated by `moon new` contains a root library package and a sample `cmd/main` package. Keep the root package transport-independent.

Future split if the code grows:

- `telnet` — constants, parser, encoder, negotiation engine.
- `telnet/options` — reusable handlers for common options.
- `telnet/fixtures` — test-only traces and RFC examples.
- `cmd/main` — demo client/server harness only, not part of core API guarantees.

## Public API sketch

- `Command` enum for TELNET commands.
- `OptionCode` newtype or alias around byte/int.
- `Event` enum for parsed output.
- `Parser` incremental state machine.
- `Encoder` helpers for data escaping and command construction.
- `Negotiator` implementing RFC 1143.
- `Policy` callbacks for accepting/rejecting options.

## Design constraints

- Accept byte chunks; never assume one read equals one TELNET message.
- Avoid allocations for ordinary data where MoonBit APIs allow slicing/borrowing.
- Preserve unknown commands/options in events so applications can log or pass through.
- Keep terminal-emulator and TCP concerns out of the core package.
